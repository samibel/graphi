01Was ist graphi und warum lohnt es sich

Ein AI-Coding-Agent, der bei jeder Frage das halbe Repo greppt und ganze Dateien neu liest, ist langsam, teuer (Tokens) und rät. graphi indexiert das Repo einmal in einen Graphen — Symbole (Funktionen, Typen, Dateien) als Knoten, Beziehungen (calls, references, defines, imports) als Kanten — und beantwortet dann Fragen wie „wer ruft das auf", „was bricht, wenn ich das ändere", „wie hängen diese zwei Funktionen zusammen" in einem gezielten Lookup, komplett auf deiner Maschine.

VorteilBedeutung für dich
Weniger TokensStatt ganze Dateien zu lesen, kommt nur das relevante Symbol + Belege zurück. graphi führt sogar ein USD-Savings-Ledger.
Exakt statt geratenDeterministischer Graph mit stabilen IDs — kein probabilistisches RAG.
VertrauenswürdigJede Kante trägt Provenienz: confidence_tier (heuristic/derived/confirmed) + Grund + Beleg (Datei:Zeile).
Schnell & frischHot-Daemon mit residentem Graph, Cold-Start P95 < 100 ms, inkrementell frisch.
Lokal-firstKein Byte verlässt die Maschine: null ausgehender Netzwerkverkehr, keine Telemetrie, CGo-frei, ein einziges Binary.

02Installation & Build

Variante A — Prebuilt Binary (schnellster Weg)

bash
# Linux/macOS – checksum-verifiziert, kein sudo, nach ~/.local/bin
curl -fsSL https://raw.githubusercontent.com/samibel/graphi/main/install.sh | sh
powershell
# Windows (PowerShell)
iwr -useb https://raw.githubusercontent.com/samibel/graphi/main/install.ps1 | iex

Variante B — Aus dem Quellcode bauen

Voraussetzung: Go 1.26+ (kein C-Toolchain nötig — der Default-Build ist CGo-frei).

bash
git clone https://github.com/samibel/graphi.git
cd graphi

# CGo-freier Build nur der CLI
CGO_ENABLED=0 go build -o graphi ./cmd/graphi

# optional auf den PATH legen
install -m755 graphi ~/.local/bin/    # oder: sudo mv graphi /usr/local/bin/

graphi version   # Version/Commit/Build-Datum prüfen
Praxis

graphi nutzt ein Go-Workspace (go.work). Beim Bauen aus dem Quellcode nicht GOFLAGS überschreiben und ggf. export GOTOOLCHAIN=auto setzen, damit die richtige Go-Version gezogen wird.

Optionale Flavors:

bash
# Breite Sprachabdeckung (257 Grammatiken, CGO, nur für vertrauenswürdige Quellen!)
CGO_ENABLED=1 go build -tags graphi_broad -o graphi-broad ./cmd/graphi

# Mit eingebetteter Web-UI
scripts/build-release-webui.sh

03Kernkonzepte in 60 Sekunden

  • Code-Graph — Repository → Knoten (Funktionen, Typen, Dateien) + Kanten (calls, references, defines, imports), mit deterministischen IDs.
  • Provenienz — jede Kante hat einen Confidence-Tier (heuristic / derived / confirmed), einen Grund und einen Beleg (Datei:Zeile). So kannst du jeder Beziehung trauen statt zu raten.
  • Eine Engine, viele Oberflächen — CLI, Unix-Socket-Daemon, MCP-stdio-Server, loopback HTTP/SSE, TUI, Web-UI und VS-Code-Extension antworten alle aus derselben Engine → Antworten sind byte-identisch und können nie auseinanderdriften.
  • Lokal-first — null ausgehender Netzwerkverkehr, keine Telemetrie, loopback-only. Beweisbar mit graphi privacy-audit.

Sprachabdeckung (Default-Tier, CGo-frei): Go (mit Type-Checker-bestätigten Kanten), TypeScript/TSX/JSX/JavaScript, Python, Java, Kotlin, C#, Ruby, PHP, Lua, C, C++, Rust, Bash, SQL sowie JSON/CSS/YAML/TOML/Markdown/HCL (bei Letzteren nur Intra-Datei). Für deine Spring-Boot-/Java-Welt ist Java also voll dabei.

04CLI-Nutzung — Schritt für Schritt

4.1 Repo indexieren

bash
mkdir -p ~/.graphi
graphi index -root . -db ~/.graphi/graph.db

Das erzeugt einen persistenten SQLite-Store (graph.db), aus dem alle weiteren Abfragen lesen.

4.2 Symbole finden (search)

Suche liefert dir die node_id, die alle strukturellen Abfragen als -symbol erwarten:

bash
graphi search -db ~/.graphi/graph.db -limit 8 OwnerController

4.3 Strukturelle Abfragen (query <op>)

bash
# Wer ruft dieses Symbol auf?
graphi query callers      -db ~/.graphi/graph.db -symbol <node_id>

# Was ruft es selbst auf?
graphi query callees      -db ~/.graphi/graph.db -symbol <node_id>

# Wo wird es referenziert?
graphi query references   -db ~/.graphi/graph.db -symbol <node_id>

# Wo ist es definiert?
graphi query definition   -db ~/.graphi/graph.db -symbol <node_id>

# Nachbarschaft (mit -depth N, gekappt auf 5)
graphi query neighborhood -db ~/.graphi/graph.db -symbol <node_id> -depth 1

Zusätzlich gibt es die typorientierten Ops: implementers, implements, overrides, subtypes, supertypes — ideal für Java-Interfaces/Vererbung.

4.4 Analyse (analyze <analyzer>)

bash
# Blast-Radius: was hängt von diesem Symbol ab? (reverse = Dependents)
graphi analyze impact -db ~/.graphi/graph.db -symbol <node_id> -direction reverse

# Aufrufpfad zwischen zwei Symbolen
graphi analyze call-chain -db ~/.graphi/graph.db -symbol <caller_id> -target <callee_id>

# Natürlichsprachliches Konzept auf Graph-Stellen abbilden
graphi analyze concept -db ~/.graphi/graph.db -symbol <root_id> -concept "rate limiting"

# Graph-Metriken: Hubs, Bridges, hohe Zentralität
graphi analyze metrics -db ~/.graphi/graph.db -symbol <node_id>

Weitere Analyzer: taint, pdg, interproc, contracts, git-history, batched (impact + call-chain + metrics in einem).

4.5 Hot-Daemon (für schnelle, wiederholte Abfragen)

bash
graphi daemon start -socket /tmp/graphi.sock -db ~/.graphi/graph.db
graphi query callers -daemon /tmp/graphi.sock -symbol <node_id>
graphi daemon stop -socket /tmp/graphi.sock

Schritt 3 — Blast-Radius: „Wer ruft Owner.getPet auf?"

bash
graphi query callers -db /tmp/petclinic.db -symbol 057c32558eb28540
json
{
  "operation": "callers",
  "outcome": "found",
  "nodes": [
    {"id":"057c32558eb28540","kind":"method","qualified_name":"owner.getPet",

Das ist der Kern-Nutzen: In einem Aufruf weißt du, dass addVisit getPet aufruft — mit Beleg (Owner.java:169) und Confidence-Tier derived. Kein Datei-Durchlesen, kein Raten.

Schritt 4 — Nachbarschaft mit bestätigter Kante

bash
graphi query neighborhood -db /tmp/petclinic.db -symbol 3bcd1cf3c507e981 -depth 1

Auszug: Die defines-Kante von OwnerRepository.java auf findByLastNameStartingWith ist confidence_tier: "confirmed" (confidence 1.0) mit Beleg OwnerRepository.java:45. „Definiert in Datei" ist beweisbar — daher der höchste Tier.

06MCP-Integration in Claude Code & andere Agenten

graphi spricht mit Agenten über MCP (stdio, JSON-RPC). Damit wird aus „der Agent liest das Repo durch" ein „der Agent fragt den Graphen".

6.1 Einrichtung in einem Befehl

bash
# 1) Graph bauen
graphi index -root . -db ~/.graphi/graph.db

# 2) graphi in alle erkannten lokalen MCP-Clients eintragen
graphi setup
#   Default --client all: Claude Code, Copilot, Cursor, Windsurf, Claude Desktop
#   Nur Claude Code:  graphi setup --client claude
#   Testlauf ohne Schreiben:  graphi setup --dry-run

# 3) claude neu starten – graphis Tools sind jetzt sichtbar

Der setup-Befehl schreibt den stdio-MCP-Eintrag idempotent, atomar und offline in die Client-Config (bei Claude Code z. B. ~/.claude.json).

6.2 Manuell für beliebige MCP-Clients

bash
graphi mcp -db ~/.graphi/graph.db

Der Handshake funktioniert nach MCP-Standard. Echter initialize-Response:

json
{"jsonrpc":"2.0","id":1,"result":{
  "capabilities":{"tools":{}},
  "protocolVersion":"2024-11-05",
  "serverInfo":{"name":"graphi-query","version":"1"}}}

Und tools/list liefert u. a. callers, callees, references, definition, neighborhood, implementers, implements, overrides, subtypes, supertypes, search — jedes mit einem JSON-Schema, das symbol (die node_id) erwartet.

6.3 Der Werkzeugkasten, den der Agent bekommt

Alle Tools sind standardmäßig read-only:

  • Struktur: callers, callees, references, definition, neighborhood, implementers, implements, overrides, subtypes, supertypes
  • Suche: search, search_semantic, compound (Cypher-artig)
  • Muster: search_ast, find_clones
  • Analyse: analyze (z. B. analyzer: "impact"), analyze_taint, analyze_pdg, analyze_interproc, analyze_contracts, analyze_githistory
  • PR-Review (experimentell): analyze_pr_risk, triage_prs, suggest_reviewers, compare_branches, critique_review, pr_comment u. a.
  • Edit (opt-in) & Readout: refactor_preview, refactor, undo, savings

6.4 Typische Agenten-Prompts (auf ein Repo angewendet)

Was du dem Agenten sagstWelches Tool feuertWarum es hilft
„Ich will IngestAll umbauen. Was hängt davon ab?"callers + analyze impact (reverse)Alle abhängigen Symbole mit Beleg, ohne halbes Repo zu lesen
„Wo wird X behandelt?"search + neighborhood + analyze conceptLandet direkt auf der richtigen Stelle
„Gibt es einen Pfad von Input-Quelle zu gefährlichem Sink?"analyze_taintFluss-sensitive Source→Sink-Pfade
„Wie riskant ist dieser Diff?"analyze_pr_risk / analyze_pr_signalsRisiko-bewerteter Diff + Signale
„Benenne price in cost um — über alle Call-Sites"refactor_preview → refactor → undoAtomare Saga mit Rollback

07Token-Ersparnis messen (Metering & Pricing)

graphi misst pro Aufruf, wie viele Tokens es gegenüber der „ganze-Dateien-lesen"-Baseline gespart hat, bepreist das mit einer eingebetteten Preistabelle (kein Netz) und führt ein durables Ledger — sogar über Daemon-Neustarts hinweg.

7.1 Die Kette

flow
meter.Record → price.Savings → cap.Apply → ledger.RecordCapped → Ledger.Readout
  • Baseline (eingefroren, versioniert): whole-file-read-v1. Der Baseline-Wert ist eine reine Funktion aus (Artefakten, Datei-Bytes) und der Version. Ältere Records bleiben ihrer Methode zugeordnet — keine stille Neuberechnung.
  • Ehrlich statt geschönt: Kann eine Baseline nicht ehrlich bestimmt werden (leere Artefakte), wird BaselineAvailable = false gesetzt und null Ersparnis gemeldet. Negative Ersparnis (graphi hat mehr gebraucht) wird roh gemeldet, nicht versteckt.
  • Anti-Gaming-Cap: engine/cap deckelt pro Op und pro Session; ein Ausreißer kann die Schlagzeile nicht aufblähen. Ein angewandter Cap wird transparent mit CapApplied markiert.

7.2 Preistabelle (engine/price/data/prices.json, Version prices-v1)

ModellInput (micro-USD/Token)Output (micro-USD/Token)
gpt-4o250010000
gpt-4o-mini150600
claude-sonnet300015000
claude-haiku2501250

(1e6 micro-USD = 1 USD. Berechnung ist exakte Integer-Multiplikation, kein Float-Rounding.)

7.3 Readout ansehen

Nach einer MCP-/Daemon-Session, die ein Ledger geschrieben hat:

bash
graphi savings -ledger <pfad>
# →  ⚡  Saved $0.42 this session  (cumulative $3.10)

MCP und CLI lesen denselben kanonischen Readout — die Zahlen sind byte-identisch.

7.4 Eigener Benchmark (wie im mitgelieferten Report)

Willst du die Ersparnis für deine eigenen realistischen Agenten-Fragen belegen: vergleiche MIT graphi (Tokens des zurückgegebenen Kontext-Bündels) gegen OHNE graphi (Whole-File-Read-Äquivalent der Dateien, die ein Agent sonst öffnen würde), mit graphis eigenem engine/meter (whole-file-read-v1) und engine/price (prices-v1). Leere-Ergebnis-Abfragen ehrlich ausschließen.

08Provenienz verstehen — warum man den Antworten trauen kann

Jede Kante trägt einen Confidence-Tier:

TierBedeutungBeispiel aus PetClinic
confirmed (1.0)Beweisbar (z. B. Type-Checker oder „definiert in Datei")defines-Kante OwnerRepository.java → findByLastNameStartingWith
derived (~0.9)Sicher aufgelöst innerhalb einer Dateicalls-Kante addVisit → getPet (Owner.java:169)
heuristicDatei/Zeile-Beleg, aber cross-file heuristisch aufgelöstdie meisten cross-file calls/imports

Wichtig: graphi erfindet nie eine Kante. Unauflösbare oder mehrdeutige Referenzen werden deterministisch verworfen und gezählt („skip+count"), nie fabriziert. Für Go läuft zusätzlich der stdlib-go/types-Checker und hebt bewiesene Kanten auf confirmed.

09Grenzen & Fallstricke (ehrlich)

Aus dem echten PetClinic-Lauf und der Doku:

  • Spring-Data-Proxy-Methoden: Interface-Methoden wie findByLastNameStartingWith (die Spring zur Laufzeit implementiert) haben oft keine callers/references-Kanten, weil der Aufruf über einen dynamischen Proxy geht. graphi meldet dann ehrlich outcome: "empty" statt zu raten. → Für DI-/Proxy-lastige Aufrufe die konkreten Implementierungsmethoden abfragen.
  • Cross-file-Auflösung ist bei Nicht-Go heuristic: Für Java/TS/Python etc. sind cross-file-Kanten heuristisch (mit Beleg), nie confirmed. Confidence-Tier immer mitlesen.
  • compare-branches nimmt keine Git-Refs: Es vergleicht zwei graphi-SQLite-Snapshots (-base/-head = Pfade zu graphi index-Outputs), nicht main/feature direkt.
  • safe-delete entfernt nur die Deklarationszeile: Bei mehrzeiligen Bodys den Diff prüfen.
  • graphi-broad (CGO) ist nicht speicher-isoliert: Nur für vertrauenswürdige/CI-Quellen. Ein C-Grammar-Crash wird von Gos Schutzmechanismen nicht abgefangen.
  • Semantische Suche ist standardmäßig AUS: search -semantic liefert ohne konfigurierten Embedder eine typisierte „unavailable"-Antwort (kein Fehler, kein Netz). Aktivieren via graphi setup-embedder + GRAPHI_EMBEDDER (Ollama loopback oder ONNX).

10Empfohlener Workflow für deinen Alltag

Für deinen Kontext (Spring-Boot-Services, tiefe Code-Analyse, MCP, Repo-Vergleiche):

  1. Einmalig pro Repo: graphi index -root . -db ~/.graphi/<repo>.db
  2. Agenten anbinden: graphi setup --client claude (oder all), claude neu starten.
  3. Vor jedem Refactor: den Agenten callers + analyze impact -direction reverse auf das Zielsymbol laufen lassen → Blast-Radius mit Belegen.
  4. Unbekannten Code verstehen: search → neighborhood → analyze concept.
  5. Bei DI/Proxy-Code: konkrete Implementierungen statt Interface-Methoden abfragen.
  6. Frisch halten: graphi indexiert inkrementell; für Dauerbetrieb den Daemon starten.
  7. Wert belegen: regelmäßig graphi savings ansehen — das macht die Token-Ersparnis sichtbar.
  8. Lokal-first beweisen (z. B. für Compliance): graphi privacy-audit.

11Schnellreferenz (Cheat-Sheet)

bash
# BAUEN
CGO_ENABLED=0 go build -o graphi ./cmd/graphi

# INDEXIEREN
graphi index -root . -db ~/.graphi/graph.db

# SUCHEN (liefert node_id)
graphi search -db ~/.graphi/graph.db -limit 8 <begriff>

# STRUKTUR
graphi query callers|callees|references|definition|neighborhood \
  -db ~/.graphi/graph.db -symbol <node_id> [-depth N]
graphi query implementers|implements|overrides|subtypes|supertypes \
  -db ~/.graphi/graph.db -symbol <node_id>

# ANALYSE
graphi analyze impact     -db ... -symbol <id> -direction reverse|forward
graphi analyze call-chain -db ... -symbol <id> -target <id>
graphi analyze concept    -db ... -symbol <id> -concept "..."
graphi analyze metrics|taint|pdg|interproc|contracts|git-history|batched -db ... -symbol <id>

# DAEMON (schnelle Wiederholabfragen)
graphi daemon start -socket /tmp/graphi.sock -db ~/.graphi/graph.db
graphi query callers -daemon /tmp/graphi.sock -symbol <id>
graphi daemon stop  -socket /tmp/graphi.sock

# MCP / AGENTEN
graphi setup [--client claude|copilot|cursor|windsurf|claude-desktop|all] [--dry-run]
graphi mcp -db ~/.graphi/graph.db      # manuell für beliebige MCP-Clients

# WEITERE OBERFLÄCHEN
graphi ui                              # Web-UI + Browser
graphi http -addr 127.0.0.1:8080 -db ~/.graphi/graph.db -root .
graphi tui  -db ~/.graphi/graph.db

# ERSPARNIS & VERTRAUEN
graphi savings -ledger <pfad>
graphi privacy-audit

# SEMANTISCHE SUCHE (opt-in)
graphi setup-embedder
export GRAPHI_EMBEDDER=ollama:127.0.0.1:11434
graphi index  --semantic -root . -db ~/.graphi/graph.db -meta ~/.graphi/meta
graphi search --semantic "wo validieren wir auth tokens" -db ~/.graphi/graph.db -meta ~/.graphi/meta
